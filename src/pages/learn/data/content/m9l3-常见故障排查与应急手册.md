实盘中永远会遇到预料之外的问题。这一节是故障排查速查表——当问题发生时，不用慌，按这个顺序排查。

## 故障 1：策略没有产生信号

**可能原因 → 排查步骤**：
1. 行情数据是否正常推送（检查行情源连接状态）
2. 数据是否满足策略的最小长度要求（如均线需要 20 根 K 线才能计算）
3. 策略参数是否被错误修改
4. 策略进程是否被暂停或崩溃

## 故障 2：委托一直不成交

**可能原因 → 排查步骤**：
1. 限价单价格与当前市价差距太大（检查盘口价差）
2. 合约已进入非主力月份，流动性极差
3. 交易所已休市（检查交易时间）
4. 该品种当日已涨停/跌停
5. 检查订单状态：是否处于 SUBMITTING（网关未响应）→ 检查网关连接

## 故障 3：账户出现预期外的持仓

**可能原因 → 排查步骤**：
1. 查看该品种全部交易记录，确认是否有遗漏的成交
2. 检查是否有手动下单（而非策略信号产生的委托）
3. 检查是否有另一个策略也在操作同一品种（策略冲突）
4. 交易所的成交回报是否延迟到达（T+1 结算的品种）

## 故障 4：实盘收益与回测严重不符

检查以下差异来源：
1. **滑点**：实盘的成交价格与回测假设的价格差多少？记录每笔交易的实际滑点。
2. **手续费**：回测中是否包含了正确的交易所手续费 + 期货公司佣金？
3. **时间对齐**：回测使用的是复权数据，实盘用的是未复权数据——主力换月时的价格跳空是否被正确处理？
4. **信号时机**：回测中假设 T 日收盘时产生信号、T 日收盘价成交（前视偏差）。实盘中通常是 T 日收盘后产生信号、T+1 日开盘成交。
5. **拒单/部分成交**：回测假设全部成交，实盘可能有 5%-10% 的委托只部分成交或完全拒单。

## 紧急停机流程（SOP）

当出现以下任一情况时，执行紧急停机：
- 单日亏损 > 5%
- 账户出现无法解释的异常持仓
- 策略发出明显不符合逻辑的信号（如连续同方向开仓 10 次）
- 行情数据出现明显错误
- 交易网关出现无法恢复的故障

**紧急停机步骤**：
1. 立即暂停所有策略的信号生成（不撤销已有委托）
2. 撤销所有未成交的挂单（撤单）
3. 评估是否需要立即平掉现有持仓（如果风险可控，可以不急于平仓以避免恐慌性滑点）
4. 记录故障发生时间、现象、初步判断原因
5. 全面排查后，逐一恢复策略（不是一次性全开，逐个验证正常后再开下一个）

## 灾备方案

| 场景 | 主方案 | 备选方案 |
|------|--------|----------|
| 主网络断线 | 有线宽带 | 4G/5G 移动热点 |
| 主服务器宕机 | 本地服务器 | 云服务器热备（阿里云/腾讯云竞价实例） |
| 行情源故障 | 主行情源（CTP） | 备用行情源（万得/聚宽/Tushare） |
| 交易网关故障 | CTP 主网关 | CTP 备用网关（端口不同） |
| 策略程序崩溃 | 自动重启脚本 | 手动重启 + 风控守护进程保护 |
| 数据库损坏 | 主库 | 从库（热备）+ 日志回放 |

:::highlight orange
最重要的故障应对能力不是「不出问题」，而是「出问题后多快能恢复」。一个能在 5 分钟内完成紧急停机 → 排查 → 恢复的策略系统，比一个从来不出问题（不存在）的系统更有实际价值。故障应对的核心纪律：先止损、再找原因、最后恢复——顺序不能错。
:::
